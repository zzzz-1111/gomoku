# 联机头像入场动画方案

## 目标

在联机对战开始时，为双方玩家增加一个“头像入场动画”：

- 双方头像从左右两侧移动到屏幕中间
- 中间短暂停顿
- 轻微闪烁或缩放强调
- 动画结束后进入正式对局

这个效果的重点是“过场感”和“对战仪式感”，不是重做整个游戏界面。

## 结论

这不是特别大的需求，但也不是纯 UI 文案修改。

如果只做一个本地播放的简单版本，工作量属于中等偏小；
如果要做到“联机双方同步、头像动态读取、动画可复用、和现有界面切换无缝衔接”，工作量会提升到中等。

推荐先做最小可行版本，再逐步增强。

## 推荐方案

### 方案 A：独立过场浮层

在主窗口上方增加一个临时的过场层，专门负责播放动画。

这个浮层只做三件事：

1. 显示双方头像、昵称、黑白方信息
2. 播放移动、停顿、闪烁动画
3. 动画结束后自动关闭，回到棋盘界面

这是最推荐的方式，因为它和棋盘逻辑解耦，不容易把联机对战主流程写乱。

### 方案 B：直接在棋盘页里做动画

把动画控件直接叠加到棋盘页面上。

优点是改动少，缺点是后续维护容易和棋盘绘制、鼠标输入、联机状态混在一起。

不推荐作为长期方案。

## 动画表现建议

建议拆成 4 段：

1. **入场**
   - 左右两边头像淡入并移动
   - 可配合昵称和黑白方标签一起出现

2. **汇合**
   - 两个头像移动到中间区域
   - 中间留出一点重叠或碰撞感

3. **强调**
   - 用一次轻微闪光
   - 或者头像缩放 1 次
   - 或者中心加一个“对战开始”字样

4. **退场**
   - 透明度降低
   - 浮层关闭
   - 进入正式对局

## 实现方式

### 1. 新增一个过场控件

建议新增一个单独的 QWidget，比如：

- `MatchIntroOverlay`
- 或 `OnlineMatchIntroWidget`

它负责绘制以下内容：

- 左侧玩家头像
- 右侧玩家头像
- 双方昵称
- 黑方 / 白方标识
- 动画文字，例如“对局开始”

### 2. 使用 Qt 动画系统

建议用这些 Qt 能力实现：

- `QPropertyAnimation`
- `QParallelAnimationGroup`
- `QSequentialAnimationGroup`
- `QGraphicsOpacityEffect`

实现上可以这样分层：

- 位置动画：控制头像从边缘移动到中间
- 透明度动画：控制淡入淡出
- 缩放动画：控制中间“闪一下”的强调感

### 3. 头像来源

头像不建议一开始就做复杂的在线上传。

建议优先使用本地默认头像，或者本地账号绑定的头像资源：

- 没有头像时，使用默认黑/白头像
- 有本地头像时，直接读取资源

这样实现风险最低，也最稳定。

### 4. 和联机流程的衔接

动画播放时机建议放在这两个节点之一：

- `GAME_START` 到达后，棋盘真正可操作之前
- 或者主机/客户端双方都完成准备后，统一进入过场

推荐由服务器发出“开始对局”信号，客户端只负责播放视觉效果，不自己决定对局开始时机。

这样可以避免联机两端状态不一致。

## 建议的数据结构

过场层至少需要这些数据：

- `hostName`
- `clientName`
- `hostAvatar`
- `clientAvatar`
- `hostSide`
- `clientSide`
- `matchMode`

如果当前项目里已经有本地登录名和黑白方信息，可以直接复用，不需要重新建一套用户系统。

## 与现有代码的关系

这个功能应该尽量只动 UI 和联机开始阶段，不碰核心落子逻辑。

建议的改动范围：

- `src/ui/mainwindow.*`
  - 负责插入过场层
  - 负责触发动画
  - 负责动画结束后的页面切换

- `src/network/game_server.*`
  - 如果需要同步双方准备完成状态，可补充开始事件

- `src/network/network_manager.*`
  - 如果需要补充昵称、头像地址或房间状态，可扩展消息字段

- `res/`
  - 放默认头像、闪光效果图标或过场背景资源

## 推荐开发顺序

### 第一阶段：最小可行版本

先实现最基础的过场效果：

- 两个默认头像
- 左右移动到中间
- 中间闪一下
- 自动进入对局

这个版本不做复杂资源管理，也不做额外网络同步。

### 第二阶段：接入真实账号昵称

把本地登录名显示到过场层里：

- 左侧显示昵称
- 右侧显示昵称
- 同时显示黑方 / 白方

### 第三阶段：接入头像资源

如果本地已有头像数据，就替换默认头像。

没有头像就继续用默认图，不影响流程。

### 第四阶段：联机同步优化

如果希望两台机器看到的动画完全一致，再补：

- 统一动画开始时间
- 统一动画时长
- 统一切换回合时机

## 风险点

### 1. 动画和输入冲突

动画播放期间必须禁用棋盘输入，否则可能出现：

- 动画还没结束就已经落子
- 画面切换和鼠标点击冲突

### 2. 联机状态不同步

如果本地先切页面、对端还没开始，就会出现状态跳变。

所以真正的“开始对局”必须由统一事件驱动。

### 3. 头像资源缺失

必须准备默认头像兜底，否则就会因为资源为空导致界面空白。

### 4. 过渡太长

如果动画时间太长，会让用户感觉卡顿。

建议总时长控制在 1.5 到 3 秒之间。

## 工作量判断

按推荐方案来看，这个需求大致可以拆成：

- 新增一个过场控件
- 接入 1 套动画
- 连接现有联机开始流程
- 补齐默认头像和昵称显示

如果不做复杂同步，整体代码量不会爆炸。
如果要做得完整、稳定、可复用，代码会明显增加，但仍然属于可控范围。

## 建议

如果你要继续做，我建议直接按下面这个顺序推进：

1. 先做一个独立过场层
2. 先用默认头像实现动画
3. 再接入本地登录昵称
4. 最后再做联机同步和资源替换

这样最稳，不会把现有联机逻辑打乱。
